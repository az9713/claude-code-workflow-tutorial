# Core Idea: Dynamic Workflow as “One Claude Window, Many Worker Agents”

The repeated pattern is:

```text
One visible Claude Code session
    → asks for a “workflow”
    → Claude decomposes the task
    → many agents inspect separate evidence slices
    → a synthesis agent merges findings
    → a verifier / devil’s advocate pass checks claims
    → final report only includes verified findings
```

The key point is **not** “ask Claude a better prompt.”
The key point is **agentic fan-out + independent verification + structured synthesis**.

The host’s central claim: most users skip the **second pass**. That second pass is what turns a fast, plausible report into something you can rely on.

# Workflow 1 — Personalized Model Upgrade Guide

## Purpose

Build a personalized guide for migrating from one model to another, for example **Claude Opus 4.7 → Opus 4.8**, based on your own Claude Code history.

Instead of reading generic release notes or hype posts, the workflow mines your own Claude Code JSONL logs and asks:

```text
Given how I actually use Claude Code,
what should I change when using the new model?
```

[TOPIC]

Goal:
Find what people are actually reporting, extract the claims, verify them,

## Gap-Fills Needed to Make It Production-Grade

### [GAP-FILL] Separate “Claim Repetition” from “Claim Verification”

A common research-agent failure:

```text
Many people repeat X.
Therefore X is true.
```

You want:

```text
Many people repeat X.
Independent evidence confirms X.
Therefore X survives.
```

Add this rule:

```text
Do not count multiple X posts as independent verification unless they contain independent evidence.
```

---

### [GAP-FILL] Add Reproduction Tier

For technical claims, classify evidence by strength:

| Tier | Evidence Type                 | Trust         |
| ---- | ----------------------------- | ------------- |
| A    | Reproduced locally            | Strongest     |
| B    | Official docs / release notes | Strong        |
| C    | GitHub issue / PR / commit    | Strong-medium |
| D    | Firsthand demo with artifacts | Medium        |
| E    | X anecdote only               | Weak          |
| F    | Repeated hype claim           | Very weak     |

This makes the report much harder to fool.

---

### [GAP-FILL] Add Budget Control

The transcript notes that this can burn tokens heavily and that the user can intervene to reduce depth.

Add:

```text
Before running full depth, ask me to choose:

Depth 1 — Quick scan:
- 20–40 claims
- 5–10 verification agents

Depth 2 — Standard:
- 75–150 claims
- 20–50 verification agents

Depth 3 — Exhaustive:
- 150+ claims
- adversarial verification
- source triangulation
```

For your use, **Depth 2** is usually the sweet spot.

# The 6-Part Framework Behind All Three

The transcript mentions a “6-part framework” but does not reveal it fully. Here is the likely reusable structure.

Marked as reconstruction:

## [GAP-FILL] General Dynamic Workflow Template

```text
1. Define objective
   What decision or artifact should this workflow produce?

2. Define corpus
   What files, logs, posts, docs, repos, folders, or APIs should be inspected?

3. Fan out analysis
   Assign many agents to independent slices of the corpus.

4. Normalize findings
   Convert outputs into a common schema:
   claim, evidence, confidence, source, recommendation.

5. Verify / adversarial pass
   Separate agents challenge the findings.
   Unsupported findings are cut or downgraded.

6. Synthesize final artifact
   Produce a report, checklist, prompt pack, skill, command, HTML page, or fix list.
```

This is the real transferable pattern.

# Minimal Implementation Standard

For every dynamic workflow you create, enforce this invariant:

```text
No final claim without:
1. source,
2. evidence,
3. confidence,
4. verification status,
5. actionability.
```

A Claude workflow without that is just parallelized hallucination. A Claude workflow with that becomes a primitive research organization.
